**Anchor:** USPTO Application **19/383,599 (BEIDNA)** · Honeycomb namespace since **1999** · FRAND/SEP-ready

## 1. Abstract
BEIDNA encodes sovereign records as a **triple-helix genome** that braids three strands—**Identity (IH)**, **Behavior (BH)**, and **Economy (EH)**—into a single machine-verifiable event. Each event emits **Tri-Header Receipts**, is anchored by a **CID**, validated by an **Admission Gate** (ethics & regulation), and safeguarded by **FR2 bounded rollback** (deterministic corrections via prefunded reserves, no history deletion). The scheme enables standard-essential adoption under FRAND.

## 2. Motivation
Legacy identity/finance systems fragment records, impede auditability, and cannot safely correct mistakes. BEIDNA unifies them via a composable encoding algebra:  
`encode | splice | fork | merge | roll | verify`.

## 3. Core Inventions
- **Triple-Helix Encoding:** IH/BH/EH headers with invariant checks (header integrity, lineage continuity, reserve sufficiency, jurisdictional compliance).  
- **Tri-Header Receipts:** concatenated & hashed → CID; line-item verifiable.  
- **Admission Gate & Score:** policy-compiled checks (ethics, AML/KYC, jurisdiction).  
- **FR2 Bounded Rollback:** time-/value-/jurisdiction-bounded reversibility via prefunded reserves.  
- **Honeycomb Namespace (since 1999):** country/industry/interest/time cells for global routing, DID binding, and market listing.  
- **Interoperability:** ISO-4217, ISO-20022, DID/VC, IPFS/HTTP gateways.

## 4. Reference Architecture
1. **Capture:** actions → BH; identity proof → IH; valuation/clearing → EH.  
2. **Mint:** policy-driven tokenization (BEIMINT) with **Admission Score**.  
3. **Anchor:** IH‖BH‖EH → hash → CID → decentralized storage.  
4. **Clear/Settle:** GX/CX/SX with FR2 reserves and dispute registry.  
5. **Evolve:** schema-versioned, crypto-agile, jurisdiction-aware.
